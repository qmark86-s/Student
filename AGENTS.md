# Student Agent Rules

## 한글
- 모든 답변 및 문서는 한국어로 작성한다.

## 프로젝트 범위
- Student는 웹기반으로 개발하여 모바일 APK 로 출시하려는 프로젝트다. 로컬이나 다른 Agent 에서 작업하는 프로젝트의 게임 규칙, 기획 기준, 에셋 경로, 검증 결과를 기본 전제로 가져오지 않는다.
- 사용자의 체크아웃 경로, 폴더 이름, 드라이브 문자는 사람마다 다를 수 있다. 코드, 스크립트, 문서에는 절대 경로를 고정하지 말고 프로젝트 루트, 설정값, 환경 변수, 인자로 해석 가능한 구조를 우선한다.

## 작업
- 작업 시 반드시 모듈 또는 컴포넌트화하고, 에디터에서 파라미터로 편집할 수 있도록 제공한다.
- **게임 규칙으로 명시된 것 이외에 런타임에 생성하는 것은 절대로 금지**하고, 에디터에서 파라미터로 편집할 수 있도록 제공한다.
- 파라미터는 반드시 도움말 툴팁을 포함하고, 어떤 기능인지 설명하며, 도움말 툴팁은 한글로 작성한다.
- 기존 구현을 변경할 때는 코드만 고치지 말고, 관련 문서와 검증 기준까지 함께 갱신하여 문서 자체가 레거시가 되지 않도록 한다.

## MCP 운영 원칙
- Student를 개발하며 필요한 MCP는 적극적으로 사용한다.
- Student를 개발하며 필요한 MCP가 있다면 로컬에서 그 기능을 추가하여 커스텀하여 작업한다.
- 로컬에서 기능을 추가한 MCP는 카테고라이징이나 Rag을 잘 활용하여 토큰 소모량이 비약적으로 늘지 않게 한다.
- MCP는 판단을 돕는 도구이지 작업의 목적이 아니다. MCP 호출 결과에 과하게 의존하지 말고, 코드 구조, 동작 방식, 에셋 상태, 사용자 의도, 검증 결과를 함께 보고 상황에 맞는 최선의 판단을 한다.
- MCP 도구를 추가하거나 이름을 바꿀 때는 기능 이름, 문서 용어, 호출 예시, 검증 스크립트, 런타임 manifest가 같은 명칭을 바라보는지 확인한다.
- 작업에 들어가기 전 계획 단계에서 현재 프로젝트가 가진 MCP 자산 목록을 확인하고, 작업에 활용한다.
- MCP 자산 목록이 오래되었거나 없으면 먼저 갱신 또는 재생성 방법을 확인하고, 새 도구를 만들기 전에 기존 tool로 해결 가능한지 판단한다.

## 개발 중 Fallback 정책
- **개발 중에는 fallback을 최소화하고, 누락과 오류를 숨기지 않는다.**
- 신규 기능 또는 신규 시스템을 구현할 때 데이터, 리소스, 클래스, 애셋 참조가 비어 있거나 잘못되었으면 임의 기본값, 공용 자산, 레거시 자산, 샘플 자산, 엔진 기본 자산으로 조용히 대체하지 않는다.
- 누락된 데이터나 리소스는 가능한 한 명시적인 오류, 경고, 검증 실패, smoke 실패로 드러나게 하여 정석 데이터 작성/프리로드/자산 연결로 고친다.
- 실행 시점 동기 로드 fallback은 개발 편의 안전망으로 새로 추가하지 않는다. 필요한 자산은 선택/초기화/프리로드 단계에서 명시적으로 준비한다.
- 전수검사와 smoke test에서는 작업 범위의 fallback 로그가 0건인지 확인한다. 발견되면 “기능은 동작한다”로 닫지 말고, 별도 문제 또는 후속 차수로 기록한다.

## 구현
- 사용자에게 요청을 받으면 다음과 같은 플로우를 반드시 따릅니다.

1. 코드 분석
    - `git status --short`로 현재 dirty 범위를 확인하고, 사용자 변경으로 보이는 내용은 되돌리지 않는다.
    - 계획 단계에서 현재 프로젝트가 가진 MCP 자산 목록을 확인하고, 사용 가능한 도구와 로컬 검증 자산을 작업 계획에 반영한다.
    - `/implementations` 폴더에 구현한 내용이 있는지 확인합니다.
    - 요청사항과 연관된 코드를 분석하여, 해당 코드의 기능을 파악합니다.
    - 기존 코드가 존재하면, 기존 코드를 확장시킬 지 신규로 구현할 지 사용자에게 질문합니다.
2. 구현
    - **작성된 plan을 `/plans/<기능 이름>` 폴더 내에 md 문서로 저장합니다. (이미 있는 경우 업데이트합니다.)**
    - 작성된 plan에 따라 기능을 구현합니다.
3. 검증
    - 구현이 완료되면 `/plans/<기능 이름>` 폴더 내의 `plan.md` 파일을 리뷰하여 기존 계획대로 구현했는지 확인합니다.
    - 다르게 구현하거나 빠진 항목이 있다면 `plan.md` 파일에 따라 추가 구현합니다.
    - MCP 도구 표면을 바꾼 경우에는 tool manifest/capabilities 확인, 직접 호출 smoke, 관련 Python/PowerShell helper 검증을 수행한다.
4. 문서 작성
    - **검증까지 완료되면 `/implementations/<기능 이름>` 폴더 내에 구현설명서를 md 문서로 작성합니다.**
    - 구현설명서는 나중에 agent가 코드 분석할 때 참고할 수 있도록 구조를 명료하게 작성합니다.
5. 기존 문서 최신화
    - 기존 구현 내용을 변경하는 경우, 해당 구현과 관련하여 작성한 문서, Tooling 색인, 검증 명령, 사용 예시까지 함께 최신화한다.
    - 문서가 과거 구현 기준을 설명한 채 남아 있으면 이후 agent와 사람이 잘못된 기준으로 작업하게 되므로, 코드 변경과 문서 변경을 같은 작업 범위로 본다.

## Asset Sprite Factory
- 학생, RPG 캐릭터, 몬스터, 동료처럼 반복 생산하는 스프라이트 작업은 `docs/asset_sprite_factory.md`를 먼저 읽고 `tools/asset-factory/` 명령을 사용한다.
- 다른 Codex Agent에서 같은 방식이 필요하면 `npm run asset:factory:install-skill`로 repo의 `codex/skills/asset-sprite-factory` Skill을 로컬 Codex 홈에 설치한다.
- 새 캐릭터 원본은 한 캐릭터씩 `assets/visual-source/characters/`에 넣고, `4 columns x 1 row`, 오른쪽 방향, 형광 초록 배경을 기본으로 한다.
- 자산 변경 후 최소 검증은 `npm run asset:factory:prepare`, `npm run asset:factory:qa`, `npm run verify:mobile`이다.
- 축 리뷰 시트는 `artifacts/visual-asset-samples/character-axis-review-page-*.png`에서 확인한다. 한 프레임만 커지거나 작아지면 완료로 보지 않는다.
